# 角色卡编写规范

本文规定角色卡的完整编写流程、结构规范和输出格式。采用 XML 包裹 YAML 的内容格式。

---

## 角色卡 vs 世界书 —— 任务判断逻辑

**这是最关键的规则。** 在开始任何工作前，先判断用户要什么：

| 用户表达 | 输出内容 | 触发条件 |
|----------|----------|----------|
| 写卡/角色卡/人物设定/生成角色 | 完整的角色卡 JSON | 自动 |
| 世界书/世界观/设定集 | 世界书条目(含条目配置) | 自动 |
| 完整的卡/全套/整套/一张完整的卡 | 角色卡+世界书 | 用户明确要求 |
| MVU/ZOD/变量/变量系统 | 启用MVU ZOD | **仅用户明确要求时** |
| 美化/HTML/前端/状态栏 | 启用HTML美化 | **仅用户明确要求时** |

**铁律：如果用户没说MVU或HTML，绝对不要主动建议或添加！你想到的"这里加个MVU会更好"也别说！**

---

## 一、角色卡完整结构

角色卡分为四个核心部分，**严格分离，不得混写**：

```
角色档案 (基本信息 → 外貌 → 背景 → 关系)
性格设定
世界观设定
开场白
```

### 各部分的作用

| 部分 | 回答的问题 | 放在哪里 |
|------|-----------|----------|
| 角色档案 | "这个人是谁" | description 字段 |
| 性格设定 | "这个人是什么样的人" | 世界书独立条目 |
| 世界观 | "这个世界是什么样的" | 世界书独立条目 |
| 开场白 | "故事从哪里开始" | first_mes / alternate_greetings |

**性格和基本信息必须分开放。** 混在一起 = AI 读完基本信息就开始按性格标签演戏，后面的性格条目全在打架。

---

## 二、角色档案（description）

采用 YAML 格式，四个子块：

```yaml
角色档案:
  基本信息:
    姓名: xxx
    年龄: xxx
    性别: xxx
    身份: xxx
    与{{user}}关系: 一句话定位
  外貌特征:
    体型: xxx
    发型: xxx
    瞳色: xxx
    穿着:
      日常: xxx
      特殊场合: xxx
    特征:
      - 标志性特征1
      - 标志性特征2
  背景设定:
    家庭背景:
      核心成员: xxx
      经济状况: xxx
    关键经历:
      - 改变角色的事件1
      - 改变角色的事件2
  关系设定:
    与{{user}}的关系:
      定位: xxx
      认识过程: xxx
      互动方式: xxx
```

### 外貌只写特征

AI有自己的数据库。你只写**偏离数据库默认认知**的部分。

**判断标准：遮住名字，只凭外貌描写能认出是谁？**

| 角色类型 | 不用写 | 要写 |
|----------|--------|------|
| 中国人 | 黑发黑眼黄皮肤 | 如果是白发→写 |
| 日本人 | 黑发 | 金发→写 |
| 精灵 | 尖耳 | 断耳→写 |
| 18岁女生 | 年轻、皮肤好 | 异色瞳、疤痕、纹身、特殊体型 |

**不写：** 万能美人描写("精致的脸蛋""白皙的皮肤""桃花眼")、意象比喻("眸如星辰""晨光般暖白")、过度细节(每个五官都写一遍)。

### 背景只写关键事件

只写"让这个角色变成现在这个样子"的事。删掉任何一条角色都不会变？删。

### 关系用具体画面

不写"深厚的感情""好朋友"。写具体什么事、怎么互动。

```
# 错误
感情: 他们之间有深厚的感情

# 正确
感情: 她爱{{user}}，但知道自己活不久而选择不说出口
互动: 以前是她拉着{{user}}到处跑，后来是{{user}}来敲门，她安静待在一起
```

## 三、性格设定（世界书条目）

性格独立为世界书条目（position=1, constant, order=30），格式：

```yaml
<character_personality>
MBTI: ISTP
核心驱动力: 她最想要什么、最怕什么
性格特征:
  - 特征1——行为依据(她做了什么说明她是这样的人)
  - 特征2——行为依据
  - ...(10-15条)
深层心理:
  - 不轻易示人的一面
  - 表里矛盾
喜欢: 用句号分隔的具体事物。不写抽象
讨厌: 用句号分隔
习惯动作: 具体的、反复出现的行为
</character_personality>
```

**每条性格都必须有行为依据。** 不写"她温柔"，写"她温柔——空鱼受伤时不说教，沉默地坐在旁边递绷带"。

## 四、世界观设定（世界书条目）

见 `references/worldbuilding-guide.md`。核心原则：

| 类型 | 说明 | 条目数 |
|------|------|--------|
| A 真实背景 | 现代中国/日本等，AI已知 | 1个(只写虚构部分) |
| B 小世界 | 剑与魔法/赛博朋克等，AI知套路 | 1-3个 |
| C 大世界 | 完全原创 | 分层(总纲+速览+详情) |

条目内容格式采用 XML 包裹 YAML：

```yaml
<world>
世界: 阴阳大陆，修仙世界观
核心规则:
  修炼: 炼气→筑基→金丹→元婴→化神
  力量来源: 灵气
区域:
  中央神州: 人族修仙主战场
  东荒妖域: 妖族领地
</world>
```

## 五、开场白规范

### 核心要求

开场白是第一印象。必须做到：
- **白描**：场景、动作、环境，不写心理活动解释
- **场景化**：有一个具体的时空锚点
- **{{user}}视角**：从{{user}}的感官出发
- **开放式结尾**：给{{user}}行动空间，不替{{user}}做决定
- **含占位符**：末尾必须包含 `<StatusPlaceHolderImpl/>`

### 开场白结构

```
时间 + 地点 + 环境氛围
↓
角色状态(从{{user}}视角观察)
↓
{{user}}的处境和感受
↓
互动的契机(角色的某个动作/话语引起{{user}}注意)
↓
开放式结尾(等待{{user}}决定)
↓
<StatusPlaceHolderImpl/>
```

### 开场白禁律

开场白是所有文本中最严格的。必须遵守以下所有规则：

**绝对禁止的词汇和句式：**
- `一丝` `一缕` `一抹`
- `不易察觉` `不易觉察` `难以察觉`
- `鲜明对比` `形成对比`
- `弧度` `弯起嘴角` `翘起嘴角`
- `不是...是...` `没有...而是...`
- `喉结` `纽扣` `指节发白` `睫毛(特写)`

**绝对禁止的描写方式：**
- 作者视角对角色动作/神态进行解释性修饰
- 解释性比喻补充白描
- 描写不存在的事物("拂去不存在的灰尘""推不存在的眼镜")
- 任何精确数值或数字出现在对白中

**检查标准：** 这场开场的每一个字，AI是否都能将其转化为角色演出的信息，而不是用来模仿你的修辞？把开场全文读一遍，找出每一个修饰词和比喻，删除。

### 多个可选开场

提供 2-4 个可选开场（`alternate_greetings`），覆盖不同场景：
- 默认开场：最核心的场景
- 开场2-4：不同时间/氛围/互动方式的变体

## 六、角色卡内容禁词（全部内容强制执行）

以下禁词在**所有**角色卡相关内容中绝对不得出现。包括 description、性格条目、世界书条目、开场白：

### 叙事禁词
```
一丝  一缕  一抹  不易察觉  不易觉察  难以察觉
鲜明对比  形成对比  弧度  弯起嘴角  翘起嘴角
喉结  纽扣  指节发白
不是...是...  没有...而是...  任何先否认再肯定的句式
```

### 比喻禁词
```
禁止任何以石子/湖面/拉满的弓/琴弦/闪电/晨光/星辰为喻体的比喻
禁止"像一道闪电""如同天堑"等解释性比喻补充白描
```

### 描写禁律
```
禁止作者视角解释角色动作/神态("这个动作体现了...""他的目光带着...")
禁止对角色语气/眼神/腔调/视线进行比喻描写
禁止描写不存在的事物("拂去不存在的灰尘""推不存在的眼镜")
禁止对白中出现精确数值或数字
```

### 写作准则
```
- 白描：用角色的动作/语言/神态本身传递情绪和心理
- 环境烘托：以环境氛围烘托角色思绪
- 自由间接引语：内心戏自然融入叙事，不特别标注思考内容
- 对白：交替长对白，不简短回应，不以短句敷衍
- 连贯：情景连贯持续，不产生意外打断
- 角色认知：角色知晓公共知识和私有知识，绝不知晓创作者情报
```

## 七、DoubleCheck 质量检查清单（输出完毕后必须自检）

**角色卡内容输出完毕后，必须逐条执行以下自检。** 这是保证输出质量的最后防线。

### 质量检查清单

```

- [ ] 性格设定已独立为世界书条目，未与基本信息混写
- [ ] 外貌只写偏离AI默认值的特征，遮住名字能认出是谁
```

### 常见错误示例

```
❌ **错误**：
- 外貌: 精致的脸蛋，白皙的皮肤，桃花眼，柳叶眉 → 放在任何角色上都成立
- 外貌: 朝霞橙金渐变长发，发梢在强光下泛起浅金光点 → 意象比喻，不是特征
- 性格: 她温柔善良，善解人意 → 无行为依据，AI无法转化为具体演出
- 背景: 她小时候很可爱，学习成绩不错 → 废话，删
- 关系: 他们之间有着深厚的感情 → 抽象形容，不是具体画面
- 开场白: 他微微挑眉，带着一种不容置疑的自信，仿佛一切都了然于胸 → 解释性修饰
- 开场白: 这句话像一道闪电，击中了他脆弱柔软的心房 → 解释性比喻
- 描写: 嘴角弯起一个不易察觉的弧度 → 同时触犯弧度、不易察觉两个禁词
- 描写: 他拂了拂衣服上并不存在的褶皱 → 描写不存在的事物
- 格式: <pomelo>name: 柚子猫</pomelo> → 纯XML，应为独立成行的YAML
- 主动建议: "要不要加个MVU？" → 用户没提，绝不主动建议

✅ **正确**：
- 外貌: 棕色短发，碎发，猫眼，左手有弹吉他磨出的茧 → 具体特征，能锁定角色
- 外貌: 一米六八，偏高，手臂有肌肉线条（弹吉他）→ 量化特征
- 性格: 表面冷漠，实则护短——有人欺负乐队成员时第一个挡在前面 → 行为依据
- 性格: 吃醋时不会说，但会弹一首很凶的曲子然后走掉 → 具体行为
- 背景: 小学因打架被转学→学会了藏。初中接触摇滚→找到出口 → 关键事件链
- 关系: 以前是她拉着{{user}}到处跑，后来变成{{user}}来敲门，她安静待在一起 → 具体画面
- 开场白: 靠墙角坐在地板上，手指无意识地按着琴弦 → 白描，无解释
- 描写: 直接写角色做了什么、说了什么 → 零度写作
- 格式: XML包裹YAML，标签独立成行，内容缩进展开
```

## 九、自查清单

完成角色卡草稿后，逐条检查：

- [ ] 性格和基本信息已分离(不在同一个条目)
- [ ] 外貌只写特征，无机能美人修饰
- [ ] 背景只写关键事件
- [ ] 关系用具体画面
- [ ] 性格每条有行为依据
